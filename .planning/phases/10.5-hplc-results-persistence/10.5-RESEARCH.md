# Phase 10.5: HPLC Results Persistence - Research

**Researched:** 2026-03-18
**Domain:** SQLAlchemy model enrichment, FastAPI endpoint augmentation, React flyout state management
**Confidence:** HIGH — all findings verified directly from source files in this repo

---

## Summary

Phase 10.5 enriches the existing `hplc_analyses` table and the `POST /hplc/analyze` endpoint with full provenance fields (calibration_curve_id, sample_prep_id, instrument_id, source_sharepoint_folder, chromatogram_data, run_group_id), then adds a result-reload path so the flyout skips SharePoint re-scan when results already exist.

The schema migration pattern is fully established: `database.py::_run_migrations()` runs `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` statements on every startup, before `Base.metadata.create_all()`. The `HPLCAnalysis` SQLAlchemy model just needs new `Mapped[]` fields to match. No Alembic, no separate migration tooling.

The flyout (`SamplePrepHplcFlyout.tsx`) currently resets all state on open and always downloads from SharePoint. Adding a "load saved results" path requires a new API endpoint (`GET /hplc/analyses/by-sample-prep/{id}`) and a state branch in the flyout: if saved results exist, render them immediately and offer a "Re-run" button.

**Primary recommendation:** Add fields incrementally to the model + migration, augment the analyze endpoint to accept and persist the new fields, add one new GET endpoint, and modify the flyout open/reset flow to check for prior results first.

## Architecture Patterns

### Pattern 1: Adding columns to existing SQLAlchemy tables

**The established flow in this repo:**

1. Add migration statement to `database.py::_run_migrations()` list:
   ```python
   "ALTER TABLE hplc_analyses ADD COLUMN IF NOT EXISTS calibration_curve_id INTEGER REFERENCES calibration_curves(id) ON DELETE SET NULL",
   "ALTER TABLE hplc_analyses ADD COLUMN IF NOT EXISTS sample_prep_id INTEGER",  # plain int, NOT FK — different DB
   "ALTER TABLE hplc_analyses ADD COLUMN IF NOT EXISTS instrument_id INTEGER REFERENCES instruments(id)",
   "ALTER TABLE hplc_analyses ADD COLUMN IF NOT EXISTS source_sharepoint_folder VARCHAR(1000)",
   "ALTER TABLE hplc_analyses ADD COLUMN IF NOT EXISTS chromatogram_data JSON",
   "ALTER TABLE hplc_analyses ADD COLUMN IF NOT EXISTS run_group_id VARCHAR(36)",  # UUID string
   ```

2. Add matching `Mapped[]` columns to `HPLCAnalysis` in `backend/models.py`:
   ```python
   calibration_curve_id: Mapped[Optional[int]] = mapped_column(ForeignKey("calibration_curves.id", ondelete="SET NULL"), nullable=True)
   sample_prep_id: Mapped[Optional[int]] = mapped_column(Integer, nullable=True)   # plain int — no FK, different DB
   instrument_id: Mapped[Optional[int]] = mapped_column(ForeignKey("instruments.id"), nullable=True)
   source_sharepoint_folder: Mapped[Optional[str]] = mapped_column(String(1000), nullable=True)
   chromatogram_data: Mapped[Optional[dict]] = mapped_column(JSON, nullable=True)  # {times: number[], signals: number[]}
   run_group_id: Mapped[Optional[str]] = mapped_column(String(36), nullable=True)  # UUID
   ```

3. Add relationship for calibration_curve (optional but consistent with WizardSession pattern):
   ```python
   calibration_curve: Mapped[Optional["CalibrationCurve"]] = relationship("CalibrationCurve")
   instrument_obj: Mapped[Optional["Instrument"]] = relationship("Instrument", foreign_keys="[HPLCAnalysis.instrument_id]")
   ```

- `backend/database.py` lines 61–124 — all prior migrations follow this exact pattern
- `backend/models.py` lines 299–353 — `CalibrationCurve` model for reference on chromatogram_data JSON and instrument_id FK

### Pattern 2: sample_prep_id is a plain integer — not a FK

**Critical constraint (from STATE.md):** `sample_preps` lives in the `accumark_mk1` PostgreSQL database (accessed via `mk1_db.py` / psycopg2). `hplc_analyses` lives in the main SQLAlchemy database. Cross-database foreign keys are not possible.

- `sample_prep_id` stores the integer `id` from `sample_preps.id` as a plain `Integer` column
- No `ForeignKey()` constraint — just `mapped_column(Integer, nullable=True)`
- Query pattern: to find all analyses for a prep, `WHERE sample_prep_id = :id`

### Pattern 3: chromatogram_data JSON shape

Established in `CalibrationCurve.chromatogram_data` (Phase 09):
```python
# Shape: {"times": [float, ...], "signals": [float, ...]}
chromatogram_data: Mapped[Optional[dict]] = mapped_column(JSON, nullable=True)
```
The flyout already produces downsampled traces via `parseChromatogramCsv` + `downsampleLTTB(raw, 800)`.
For `hplc_analyses`, store the same shape. For blends with multiple analyte traces, store per-analyte:
```python
# Option A: single trace (non-blend)
{"times": [...], "signals": [...]}
# Option B: per-analyte (blend)
{"BPC": {"times": [...], "signals": [...]}, "GHK": {"times": [...], "signals": [...]}}
```
Keep it flexible — store whatever trace data is available.

### Pattern 4: run_group_id for blend run grouping (HRES-05)

For blend preps, `runAllAnalyses()` in the flyout makes N separate calls to `POST /hplc/analyze` (one per analyte/component). Currently each produces an independent `hplc_analyses` row with no link between them.

**Solution:** Generate a UUID in the flyout before the loop, pass it as `run_group_id` in each request:

Frontend (flyout):
```typescript
// Before the for-loop in runAllAnalyses()
const runGroupId = isBlend ? crypto.randomUUID() : undefined

// In each call:
const res = await runHPLCAnalysis({
  ...existingFields,
  run_group_id: runGroupId,
  sample_prep_id: prep.id,           // integer id from sample_preps
  instrument_id: prep.instrument_id ?? undefined,
  source_sharepoint_folder: match.folder_name,
  chromatogram_data: chromTraceForAnalyte,
})
```

Backend receives and stores it. Querying a blend group: `SELECT * FROM hplc_analyses WHERE run_group_id = :uuid`.

### Pattern 6: New GET endpoint — analyses by sample_prep_id

Add `GET /hplc/analyses/by-sample-prep/{sample_prep_id}` to `main.py`:

```python
@app.get("/hplc/analyses/by-sample-prep/{sample_prep_id}", response_model=list[HPLCAnalysisResponse])
async def get_analyses_by_sample_prep(
    sample_prep_id: int,
    db: Session = Depends(get_db),
    _current_user=Depends(get_current_user),
):
    analyses = db.execute(
        select(HPLCAnalysis)
        .where(HPLCAnalysis.sample_prep_id == sample_prep_id)
        .order_by(desc(HPLCAnalysis.created_at))
    ).scalars().all()
    return [_analysis_to_response(a, db) for a in analyses]
```

Extract a helper `_analysis_to_response(analysis, db) -> HPLCAnalysisResponse` to avoid code duplication with the existing GET by id endpoint.

### Pattern 7: Flyout reload path (HRES-06)

Currently `SamplePrepHplcFlyout.tsx` always resets state on open (`useEffect([open, prep.id])`). The new flow:

1. On open: call `GET /hplc/analyses/by-sample-prep/{prep.id}` to check for saved results
2. If results exist (most recent run): populate `analyteResults` / `result` state directly, skip SharePoint download, show a "Re-run" button
3. If no saved results: proceed with existing SharePoint flow unchanged
4. "Re-run" button clears saved state and triggers the existing `loadPeakData()` + `runAllAnalyses()` flow

State to add:
```typescript
const [savedResults, setSavedResults] = useState<HPLCAnalysisResult[] | null>(null)
const [loadingResults, setLoadingResults] = useState(false)
```

The `useEffect` on open changes from "always reset + load" to "reset + check DB first, then conditionally load from SharePoint."

### Pattern 8: Sample prep status → hplc_complete (HRES-07)

After a successful analysis run in the flyout, call `PATCH /sample-preps/{prep.id}` with `{ status: 'hplc_complete' }`.

**Required change in mk1_db.py:** Add `hplc_complete` as an allowed status. The `update_sample_prep()` function's `allowed` set and `SamplePreps.tsx` `STATUSES` array need `hplc_complete` added.

The `sample_preps.status` column is `VARCHAR(30)` with default `awaiting_hplc`. No DDL change needed — just add it to the allowed values and UI list.

Frontend (after `runAllAnalyses()` completes successfully):
```typescript
await updateSamplePrep(prep.id, { status: 'hplc_complete' })
```

## Common Pitfalls

### Pitfall 1: sample_prep_id FK constraint
**What goes wrong:** Adding `ForeignKey("sample_preps.id")` to `HPLCAnalysis` — this table doesn't exist in the same DB.
**Why it happens:** Forgetting the two-database architecture.
**How to avoid:** `mapped_column(Integer, nullable=True)` — no ForeignKey wrapper. No relationship. Document it with a comment.

### Pitfall 2: Migration order — ALTER before create_all
**What goes wrong:** Adding only the `Mapped[]` field to models.py without adding the migration statement. `create_all` handles NEW tables but doesn't ALTER existing tables. On a live database with existing `hplc_analyses` rows, the column won't exist.
**How to avoid:** Always add both the migration statement AND the model field. Migration runs first in `init_db()`.

### Pitfall 3: HPLCAnalysisResponse missing new fields
**What goes wrong:** Adding fields to the model and request but forgetting to add them to `HPLCAnalysisResponse`. The existing GET endpoint at line 2950 builds the response manually — it will silently omit new fields.
**How to avoid:** Update `HPLCAnalysisResponse`, update the `return HPLCAnalysisResponse(...)` call in `run_hplc_analysis()`, and update the similar construction in `get_hplc_analysis()`.

### Pitfall 4: run_group_id collisions on re-run
**What goes wrong:** If user re-runs analysis from the flyout, a new `run_group_id` is generated — old rows with the old group ID still exist. This is correct behavior (old run is historical), but the DB may accumulate multiple run groups per prep.
**How to avoid:** The "load saved results" path should fetch the most recent run group. When re-running, generate a new UUID. No cleanup of old rows needed (they're audit history).

### Pitfall 5: Flyout state reset timing
**What goes wrong:** The existing `useEffect([open, prep.id])` resets everything. If the DB check for saved results happens after the reset, it will clear the loaded results.
**How to avoid:** The DB check and result population must happen within the same open handler sequence. Load saved results first, only trigger SharePoint flow if no saved results found.

### Pitfall 6: Blend chromatogram data — which trace per analyte
**What goes wrong:** Blend preps have multiple analyte traces (e.g., `PB-0051_Inj_1_GHK.dx_DAD1A`, `PB-0051_Inj_1_BPC.dx_DAD1A`). The flyout already filters `displayChromTraces` by `activeAnalyte`. But storing them: need to pick the right trace per analyte call.
**How to avoid:** When passing `chromatogram_data` to each analyte's `runHPLCAnalysis()` call, filter `chromTraces` by analyte label (same logic as `displayChromTraces`). If no analyte-specific trace, store the combined/first trace. Accept null if no trace available.

## Open Questions

1. **Chromatogram storage for blends**
   - What we know: Blend preps have per-analyte traces; flyout already filters by label
   - What's unclear: Whether to store a single JSON per `hplc_analyses` row (per-analyte, since each row is per-analyte) or store all traces in one row
   - Recommendation: Since each `hplc_analyses` row IS per-analyte (one row per component in a blend), store only that analyte's chromatogram trace as `{times: [], signals: []}` — not a nested object. Simple, consistent with CalibrationCurve pattern.

2. **run_group_id for non-blend single-analyte preps**
   - What we know: Single preps produce one `hplc_analyses` row; no grouping needed
   - What's unclear: Whether to still generate a run_group_id for single preps (for consistency / potential future use)
   - Recommendation: Generate run_group_id for ALL runs (blend and single). Single-row groups are valid. Cost is zero.

3. **Reload UX — full results or summary?**
   - What we know: Saved results lack the raw injection/peak data (that's only in SharePoint files + current session)
   - What's unclear: Whether the AnalysisResults + CalculationVisuals display components need raw injection data that isn't in DB
   - Recommendation: `raw_data` column already stores `{"injections": [...]}` (set at line 2813). The response currently omits it. Add `raw_data` to HPLCAnalysisResponse so the flyout can reconstruct `parseResult`-like state from DB. If too complex, initial scope: show results in read-only mode without peak table.

4. **Status update caller**
   - What we know: `updateSamplePrep()` API function exists, `update_sample_prep()` in mk1_db.py handles PATCH
   - What's unclear: Whether the flyout should update status or a separate explicit user action
   - Recommendation: Auto-update to `hplc_complete` when analysis succeeds. Match the `hplc_analyses` row creation event. User can still manually change status back via the SamplePreps table dropdown.
