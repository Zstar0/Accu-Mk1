# Accumark 1.0 Platform Release — Deploy Guide (2026-06-16)

**Release scope:** the variance-testing pipeline end-to-end (per-vial assignment,
lab override, COA variance series, customer variance report), plus the
sub-sample / native-analyses / Manage-Analyses / Replace-Analyte / customer-remarks
work that makes Accu-Mk1 the primary LIMS. Branded **Accumark 1.0**.

> Deploy is run from the `accumark-deploy` skill playbooks (GHCR services via
> `scripts/deploy.sh`, WordPress via `deploy.py`). This doc is the cross-service
> coordination layer: versions, ordering, pre-deploy data, and the COABuilder hold.

## ⚠ COABuilder is HELD

COABuilder `feat/coa-identity-na-variance` (2.26.0) carries the **variance-COA
mean-based certification model**, which is **parked awaiting a lab decision**
(`project_variance_coa_model_parked`; the digital-COA contradiction is live).

**Do NOT deploy COABuilder as part of this release unless the lab has signed off.**
Decide before merging it to master — merging stages it for the stack deploy.

**Cross-service dependency to confirm:** the customer Variance Report is
*server-authored by COABuilder* and forwarded via IS → WP. Shipping the
Accu-Mk1/IS/WP variance surfaces while COABuilder is held may leave the variance
report non-functional (or rendering against an old model) in prod. Confirm what
the deployed COABuilder version actually produces before enabling variance for
real customers.

## Pre-deploy data / config (do FIRST — silent failures otherwise)

1. **Integration Service `.env`:** set **`SENAITE_VARIANCE_PROFILE_UID`** to the
   SENAITE `sample_variance` profile UID. Missing → variance orders never attach
   the variance profile (no error, just wrong AR composition).
2. **WordPress (WooCommerce):** create the **"Variance" Shadow product** and the
   `wc_test_services` entry (name contains "variance") — `project_variance_launch_wp_data`.
   The customer variance report needs **no page** (it renders as a portal section).
3. **`JWT_SECRET` must match** across WordPress (`ACCUMARK_JWT_SECRET`), Integration
   Service (`JWT_SECRET`), COABuilder (`JWT_SECRET`). Mismatch → COA generation
   silently fails. (Not rotating this release unless you choose to — just verify.)
4. **Variance beta gating (optional):** the wizard/report is gated by the WC
   product publish-status + per-user `_accumark_variance_tester` flag. Keep the
   product as Draft + flag testers for a limited beta, or publish for GA.

## Deploy order (skill: stack dependency order)

**Integration Service → COABuilder → Accu-Mk1 → WordPress.** Verify each health
check before the next. If COABuilder is held, skip it (the others don't hard-depend
on the *new* COABuilder for non-variance COAs).

1. **Integration Service → 1.0.0** — `bash scripts/deploy.sh`; run alembic if new
   migrations; `curl …:8000/v1/healthz` + `/v1/readyz`. Confirm `SENAITE_VARIANCE_PROFILE_UID`.
2. **COABuilder → 2.26.0** — **HELD** (skip unless lab-approved). `…:5000/health`.
3. **Accu-Mk1 → 1.0.0** — `bash scripts/deploy.sh` (web/backend first), then cut the
   desktop release. `curl https://accumk1.valenceanalytical.com/api/health`.
4. **WordPress (wpstar 2.26.0)** — `python deploy.py` (tar-based; NEVER DevKinsta
   "Push to Production"). `curl -s -o /dev/null -w "%{http_code}" https://accumarklabs.com/wp-json/` (200).

## Merge-to-master (pre-deploy)

All four are clean fast-forwards. Merge the shipping three before deploy:

```
# per repo, on master:
git checkout master && git merge --ff-only subsample-features        # Accu-Mk1, accumarklabs, integration-service
git checkout master && git merge --ff-only feat/coa-identity-na-variance   # COABuilder — ONLY if lab-approved
```

Tag + GitHub release per service at deploy time (the deploy scripts handle the tag
+ New Relic marker; don't skip).

## Rollback

Every GHCR service auto-rolls-back on a failed health check; manual rollback reads
`/root/<service>/.deploy/previous_version`. WordPress rolls back from the pre-deploy
tar. See the `accumark-deploy` skill.
