# Accumark 1.0 — Post-Deploy / Backfill Checklist

*Compiled 2026-06-19 for the 1.0 release candidate. Run order top-to-bottom after the code deploy of all four repos. Use the `accumark-deploy` skill for deploy mechanics + ordering + JWT.*

---

## 0. Runs AUTOMATICALLY on deploy — no action

- **Accu-Mk1 schema** — `database.py::_run_migrations()` executes a list of idempotent `ALTER TABLE … ADD COLUMN IF NOT EXISTS` on every backend startup (no alembic, no version table). Covers `result_type`/`result_options` (lines 621-622), the lims_* tables, `audit_logs`, etc.
- **Accu-Mk1 `variance_capable` seed** — run-once guarded ADD COLUMN + `UPDATE analysis_services SET variance_capable=TRUE WHERE keyword IN ('PH-DETERM','Benzyl_Alcohol_Assay','FILL-NET-CONTENT')` (database.py ~751-761). **Automatic** — do NOT run by hand.

> Mk1 has no migration head; `repo_shas.Accu-Mk1` is the schema pointer.

## 1. Accu-Mk1 — `POST /analysis-services/sync`  ⟵ REQUIRED, manual

Seeds `analysis_services.result_options` from SENAITE so **sub-sample selection-type result inputs render a dropdown** (Sterility/PCR, HPLC-ID). The startup migration adds the *column* but leaves it empty for services that predate the seed logic; the parent reads SENAITE live and masks the gap, so without this the sub-sample inputs are free-text.

- Authenticated `POST /analysis-services/sync` against the prod Mk1 backend.
- Additive / local-wins / idempotent (safe to re-run). Also adds any new SENAITE services.
- **Verify:** `SELECT keyword, result_options FROM analysis_services WHERE result_type='select';` → STER-PCR / PCR-FUNGI / PCR-BACTERIA (0=Not Detected,1=Detected) + HPLC-ID (1=Conforms,0=Does Not Conform).
- See memory `architecture_subsample_result_options_seed`.

## 2. Integration-Service — alembic  ⟵ verify only (no new migrations this release)

IS alembic does NOT auto-run (Dockerfile CMD is uvicorn only; deploy.ps1 doesn't `alembic upgrade head`). **This release adds NO new migration** (`origin/master..subsample-features -- migrations/versions/` is empty; the 3 release commits are code-only).

- Confirm prod is at head: `SELECT version_num FROM alembic_version;` (against `accumark_integration`). If behind from a prior release, `alembic upgrade head` against the running container.
- **Restart IS after deploy** — no `--reload`; `docker restart integration-service` or it serves stale code (this bit us with the additional-COA variance bug).
- See memory `architecture_db_migrations`.

## 3. accumarklabs (WP) — store data + COA backfill  ⟵ manual

### 3a. "Variance" Shadow WC product + `wc_test_services` entry — confirm present
Variance line items price per-data-point by scanning `wc_test_services` for a row whose **name contains "variance"** with `product_id` → a published Variance shadow product. **Without it, variance line items price at $0.** The existing shadow product is shared across panels (incl. BW) — likely already in prod from the earlier variance work; **confirm it exists** (and in DevKinsta for local UAT). The customer Variance Report needs **no WP page** (it's a `/portal/?section=variance-report` section). See memory `project_variance_launch_wp_data`.

### 3b. COA result-breakdown backfill — REQUIRED for existing COAs
The `/portal/coas/` result-breakdown reads the local snapshot `wp_accumark_client_peptides.test_results_json`. **New COAs populate `specification`+per-vial `variance` automatically on notify; existing rows show `specification: null` / no vial table until backfilled.**
- Recreate/adapt the dev script (`_backfill-coa-specs.php`, self-deleting — not in repo) for prod: one `IntegrationService::verify_code()` call per row → re-shape tests → merge variance from order meta → UPDATE `test_results_json`. Idempotent. (Local run: 117 updated.)
- Other tracked backfill scripts live in `wp-content/themes/wpstar/scripts/` (e.g. `backfill-client-peptides.php`) — pattern reference.
- See memory `project_coa_spec_variance_backfill`.

### 3c. mu-plugin `accumark-hardening.php` — manual, if not already live
Prod hardening lives in a mu-plugin, NOT the theme; `deploy.py` is theme-only, so mu-plugins deploy manually. Confirm it's present in prod `wp-content/mu-plugins/`. (Currently untracked in the repo — separate concern, not gated on this release.) See memory `architecture_wp_security_hardening`.
