# COA Source Roll-Up with Override Panel — Design

**Date:** 2026-06-02
**Status:** Draft — not started.
**Scope:** Decide *which sub-sample's result* a parent's COA reports for each analyte when results live across the parent AR and one-or-more sub-sample ARs (variance HPLC, endo on the endo vial, sterility on the ster vial). Auto-rolls up the unambiguous case; surfaces an override panel and freezes a per-generation manifest for the ambiguous case. Adds a permanent "how was this COA derived" view on the parent sample-details page.
**Repos touched:** `Accu-Mk1` (backend + frontend), `coabuilder` (input contract extension).
**Predecessor phases:** `feat/sub-samples` (sub-sample ARs), `feat/vial-assignment-step` (HPLC/ENDO/STER/XTRA roles), the worksheet-variance work in `2026-06-02-worksheet-variance-grouping-design.md` (variance set membership + lock state).
**Successor phase:** Variance Addon COA section consumes locked variance sets via the same source-resolution machinery — separate spec, may be folded into this one if the contract proves identical.

---

## Goals

- **One COA per parent.** Customer sees `BW-0013.pdf`, not per-sub PDFs. The parent AR remains the customer-facing identity; sub-sample identities are internal.
- **Zero-touch happy path.** When exactly one vial in the family has a verified result for a given analyte (the typical case: endo on the endo vial, ster on the ster vial, single-vial HPLC on the parent), the COA picks that result without prompting.
- **Explicit choice when ambiguous.** When >1 vial has a verified result for the same analyte (variance HPLC, retest replays, the same analysis run on two subs by error), block COA generation and force a manager pick. No silent first-wins.
- **Per-generation manifest is immutable.** Every published COA freezes "which result, from which vial, with what value" at the time of publish. Pins evolve; the v2 manifest still shows what v2 reported even after v3 ships.
- **Single-screen derivation view.** From the parent's sample-details page a user can answer "where did each row of this COA come from, and why?" in two clicks per row, for *any* historical generation.
- **Regen-and-republish stays a one-button flow.** The existing `Regen & Republish` affordance on `PublishedCOACard` re-runs the roll-up with current pins — no separate "publish-from-builder" UX.
- **Variance set integration.** When a parent has a locked variance set (per the variance-grouping spec), the locked set acts as a soft default — the COA picks the variance-set summary value (mean or selected representative) instead of forcing an individual-vial pin. Manual pins still override.

## Non-goals

- **No customer-facing per-analyte source disclosure.** The published COA reports values; it does NOT print "Identity was measured on sub-vial BW-0013-S02." Source provenance is internal-only. (A future regulatory ask might revisit.)
- **No COA builder wizard.** Generation stays one button. The override panel is for *fixing the inputs to the roll-up*, not for hand-assembling the COA row-by-row. (Option C from the design discussion; rejected as too high-friction for the 90% case.)
- **No SENAITE schema changes.** Results stay on their respective ARs in SENAITE. Mk1 layers the source-selection model on top.
- **No automatic aggregation of multiple results into one COA value in v1.** Aggregation (mean / median / pooled value) is on the roadmap as `mode = aggregate` (see *Open questions*) but ships in a follow-on phase, gated behind the variance-addon work.
- **No multi-COA generation in one click.** Each parent COA is generated by its own button press. Bulk generation is unchanged.
- **No retroactive backfill of source pins for already-published COAs.** Existing generations stay as they are — they don't get a manifest reconstructed after the fact. The COA Sources panel for those generations shows "manifest unavailable (pre-feature)" and falls back to the existing analysis snapshot.

## Architecture

### Roll-up algorithm

At COA generation time (`/coa/{sample_id}/generate`), for each analyte the parent's order requires:

```
def resolve_source(parent_sample_id, analyte_keyword) -> SourceDecision:
    candidates = gather_candidates(parent_sample_id, analyte_keyword)
    # candidates: list of {source_sample_id, source_analysis_uid, value, unit, state, reportable, variance_in_set}

    candidates = [c for c in candidates
                  if c.state in ('verified', 'published')
                  and c.reportable]

    if not candidates:
        return SourceDecision.blocking('missing', analyte_keyword)

    if len(candidates) == 1:
        return SourceDecision.auto(candidates[0])

    # > 1 candidate
    pin = get_pin(parent_sample_id, analyte_keyword)
    if pin and pin.mode == 'pin':
        # Match the pin to an actual candidate (guard against stale pin)
        match = next((c for c in candidates
                      if c.source_sample_id == pin.source_sample_id
                      and c.source_analysis_uid == pin.source_analysis_uid),
                     None)
        if match:
            return SourceDecision.pinned(match, pin)
        # Pin no longer matches a live candidate — surface as ambiguous
        return SourceDecision.blocking('stale_pin', analyte_keyword,
                                       reason=f'pinned source {pin.source_sample_id} no longer has a reportable verified result')

    # Locked variance set is a soft default for HPLC analytes
    if is_locked_variance_analyte(analyte_keyword) and locked_variance_set_complete(parent_sample_id):
        rep = variance_set_representative(parent_sample_id, analyte_keyword)
        return SourceDecision.variance_set(rep)

    return SourceDecision.blocking('needs_decision', analyte_keyword,
                                   candidates=candidates)
```

Generation succeeds only if every analyte resolves. Any blocking decision raises a structured error that the frontend renders in the COA Sources panel with a "needs decision" badge per unresolved analyte. The user clicks each one, picks the source (or accepts the variance-set default), and re-runs generate.

**Candidate gathering** reads:
- The parent AR's analyses.
- Every sub-sample AR's analyses, joined via `lims_sub_samples` → parent.
- The `reportable` flag on each analysis instance (new per-analysis boolean; default `TRUE`. See *Data model*.)

**Why parent AR is part of the candidate pool.** Some analytes legitimately live on the parent (single-vial HPLC, water content). The parent is just "vial 1 of the family" from the roll-up's perspective.

### Two-layer persistence

**Pins table** answers "what should the roll-up use *next* time?" Mutable, edited by managers, audit-logged via `SampleActivityLog`.

**Manifest table** answers "what did v2 actually publish?" Written once per generation, immutable forever. Linked 1:N to `explorer_coa_generations` (existing table — current generations get a row per analyte in `coa_generation_sources`).

Both tables share the `(analyte_keyword, source_sample_id, source_analysis_uid)` triple as the unit of identity. Diff between pins and manifest tells you "what changed between v2 and v3 source selections."

### Override panel placement

A new **COA Sources** card on the parent sample-details page (parent ARs only — sub-sample pages don't show it). Sits between the existing Generated/Published COA cards and the Analysis table. Two modes:

**Current draft mode** (when a draft COA exists or generation has been attempted):
- One row per analyte the order requires
- Columns: Analyte · Reported value · Sourced from · Why (auto / pin / variance-set / blocking)
- Blocking rows render with an amber "needs decision" chip and an inline "Pick source" button
- Resolved rows show their source vial as a link and a quiet "change source" affordance

**Historical mode** (when viewing a published generation):
- Same table, read-only, fed by that generation's frozen `coa_generation_sources` rows
- Header shows "v2 — published 2026-05-31 by Forrest — superseded by v3"
- Drawer-per-row shows the candidates that were available *at that generation's time*, with the chosen one highlighted

Generation versioning UI (v1/v2/v3 tabs above the panel) reuses whatever exists today for COA history; the Sources card slot in just changes contents when the active generation changes.

### Pin-edit drawer

Click any row → drawer slides in from the right (matching the existing flyout pattern in `SampleDetails.tsx`):

```
┌── Identity (HPLC) — pick source ──────────────────────────────┐
│ Currently: auto                                                │
│                                                                │
│ Candidates                                                     │
│ ○ BW-0013      (parent)  · 96.2%  · verified 2026-05-30        │
│ ● BW-0013-S02  (vial 3)  · 98.55% · verified 2026-06-01        │
│ ○ BW-0013-S03  (vial 4)  · 97.8%  · verified 2026-06-01        │
│                                                                │
│ Pin source: [ BW-0013-S02       ▾ ]                            │
│ Reason: [____________________________]  (optional)             │
│                                                                │
│ [ Cancel ]                          [ Pin source ]             │
└───────────────────────────────────────────────────────────────┘
```

Reason is optional but strongly recommended — it shows up in the COA Sources table's "Why" column and in the activity log entry.

### Regen-and-republish flow

The existing `Regen & Republish` button on `PublishedCOACard` (and its additional-COA equivalent on `AdditionalCoaCard`) re-runs the generate path. With this design that means:

1. Manager edits pins via the drawer; each save writes `coa_result_pins` + an activity log row.
2. Manager hits `Regen & Republish`.
3. Backend reruns the roll-up using *current pins*. If anything blocks, regen fails fast with the structured error → COA Sources panel surfaces unresolved rows.
4. On success: new `explorer_coa_generations` row, fresh `coa_generation_sources` snapshot, new PDF, new verification code, prior generation marked `superseded` (existing behavior).
5. Activity log entry: "Forrest republished primary COA — sources changed: Identity-HPLC swapped to BW-0013-S02."

The activity-log entry diffs the new manifest against the previous published manifest, listing only the rows that *changed source*. (Value changes that come from the same source — e.g. a tech re-entered a result — aren't called out separately; the regen action itself implies a refresh.)

## Data model

Three additive surfaces. Two new Mk1-local tables, one new column on the existing SENAITE-mirror analysis table (or its Mk1-side cache, whichever the COA path reads — see *Open questions §3*).

### `coa_result_pins`

```sql
CREATE TABLE IF NOT EXISTS coa_result_pins (
    id                    SERIAL PRIMARY KEY,
    parent_sample_id      TEXT NOT NULL,
    analyte_keyword       TEXT NOT NULL,
    -- Mode controls how the roll-up consumes this row.
    --   'pin'           → use the named source_sample_id + analysis_uid
    --   'auto'          → explicit "let the resolver decide" (used to clear a pin
    --                     without deleting the row; preserves the audit trail)
    --   'variance_set'  → use the variance-set representative (mean, etc.)
    mode                  TEXT NOT NULL CHECK (mode IN ('pin', 'auto', 'variance_set')),
    source_sample_id      TEXT,   -- required when mode='pin', null otherwise
    source_analysis_uid   TEXT,   -- required when mode='pin', null otherwise
    reason                TEXT,
    pinned_by_user_id     INTEGER REFERENCES users(id),
    pinned_at             TIMESTAMP NOT NULL DEFAULT NOW(),
    UNIQUE (parent_sample_id, analyte_keyword)
);

CREATE INDEX IF NOT EXISTS ix_coa_result_pins_parent
    ON coa_result_pins (parent_sample_id);
```

**Unique on (parent, analyte)** — one pin per analyte per parent, updated in place. Audit history lives in `SampleActivityLog`, not in this table. (Alternative: append-only with `superseded_at` — rejected; activity log already covers this and the panel only reads the current state.)

### `coa_generation_sources`

```sql
CREATE TABLE IF NOT EXISTS coa_generation_sources (
    id                    SERIAL PRIMARY KEY,
    generation_id         INTEGER NOT NULL
                           REFERENCES explorer_coa_generations(id) ON DELETE CASCADE,
    analyte_keyword       TEXT NOT NULL,
    source_sample_id      TEXT NOT NULL,
    source_analysis_uid   TEXT NOT NULL,
    -- Frozen snapshot. The published PDF's value for this analyte —
    -- not re-derived from SENAITE at read time. Survives any later
    -- result edit on the source AR.
    result_value          TEXT,
    result_unit           TEXT,
    -- Audit: how many candidates were available at generation time.
    -- If candidates_count > 1 the row was either pinned or variance-set;
    -- the resolution mode tells you which.
    candidates_count      INTEGER NOT NULL,
    resolution_mode       TEXT NOT NULL CHECK (resolution_mode IN
                            ('auto', 'pin', 'variance_set', 'stale_pin_fallback')),
    UNIQUE (generation_id, analyte_keyword)
);

CREATE INDEX IF NOT EXISTS ix_coa_generation_sources_gen
    ON coa_generation_sources (generation_id);
```

**`resolution_mode='stale_pin_fallback'`** — defensive: if a pin matched a candidate at generation time but the candidate was withdrawn between rows being written, this records that the generator used the pin's frozen value even though the live AR no longer has a matching analysis. Should be vanishingly rare; included so we can detect it in audit.

### `analyses.reportable` (or equivalent on the Mk1-side cache)

```sql
ALTER TABLE analyses
    ADD COLUMN IF NOT EXISTS reportable BOOLEAN NOT NULL DEFAULT TRUE;
```

Per-analysis-instance boolean. Default `TRUE` matches today's implicit behavior (every verified result is fair game). A tech / manager toggles `FALSE` to exclude an analysis from the candidate pool — e.g. a failed variance run that should not be considered for the COA.

The flag lives on the analysis *instance*, not the analysis *service*. Excluding "the HPLC ID run on BW-0013-S03" does not exclude "HPLC ID" globally.

**Why this lives on the analysis row and not the pin row.** The reportable flag is a property of "is this individual result fit to report?", asked by techs at the bench. Pins are a manager judgment about *which fit-to-report result to choose*. Two different decisions; two different places.

If the existing schema can't take a new column on `analyses` (SENAITE-side), we mirror to a Mk1 `analysis_reportable` sidecar with `(sample_id, analysis_uid)` as the key. Tracked in *Open questions §3*.

## API endpoints

```
GET    /api/coa/{parent_sample_id}/sources
       → for the current draft COA. Body:
         {
           parent_sample_id,
           rows: [
             {
               analyte_keyword,
               analyte_display_name,
               resolution: 'auto' | 'pin' | 'variance_set' | 'blocking',
               blocking_reason: 'missing' | 'needs_decision' | 'stale_pin' | null,
               chosen: { source_sample_id, source_analysis_uid, value, unit } | null,
               candidates: [
                 { source_sample_id, source_analysis_uid, value, unit, state,
                   reportable, in_variance_set }
               ],
               pin: { mode, source_sample_id, source_analysis_uid, reason,
                      pinned_by, pinned_at } | null
             }
           ]
         }

GET    /api/coa/{parent_sample_id}/sources/{generation_number}
       → same shape, read from coa_generation_sources for the historical
         generation. resolution is the frozen mode; candidates is the live
         state at generation time (audit snapshot persisted alongside the
         row in a denormalized `candidates_snapshot` JSON column on
         coa_generation_sources — small enough to inline).

PUT    /api/coa/{parent_sample_id}/pins/{analyte_keyword}
       body: { mode, source_sample_id?, source_analysis_uid?, reason? }
       → upserts coa_result_pins. Returns the new pin row.
         Emits SampleActivityLog: "X pinned analyte Y to source Z because R".

DELETE /api/coa/{parent_sample_id}/pins/{analyte_keyword}
       → deletes the pin row (equivalent to mode='auto' UPSERT but cleaner
         semantics for "I no longer want this row in the table at all").

PUT    /api/analyses/{analysis_uid}/reportable
       body: { reportable: bool, reason?: string }
       → toggles the analyses.reportable flag. Activity-log entry on flip.

POST   /api/coa/{parent_sample_id}/generate
       → existing endpoint, behavior extended:
         - runs the resolver before invoking COABuilder
         - writes coa_generation_sources rows on success
         - returns structured error { code: 'unresolved_sources',
           unresolved: [analyte_keywords] } on any blocking decision;
           frontend surfaces these in the panel
```

## Backend

### Resolver location

New module `coa/source_resolver.py` (or fold into `coa/builder_bridge.py` if that exists). Pure function: takes `(parent_sample_id, db)`; returns a list of `SourceDecision` objects + a `blocked: list[BlockedReason]`. No side effects.

`/coa/.../generate` calls the resolver up front. On `blocked` → HTTP 422 with the structured error. Otherwise proceeds to COABuilder with the resolved sources embedded in the input payload.

### COABuilder input extension

COABuilder currently reads the parent AR + its analyses. The contract extension:

```jsonc
{
  "sample_id": "BW-0013",
  // existing fields ...
  "result_sources": {
    "IDENTITY_HPLC": { "source_sample_id": "BW-0013-S02", "value": "98.55", "unit": "%" },
    "ENDOTOXIN":     { "source_sample_id": "BW-0013-S01", "value": "<0.5", "unit": "EU/mg" }
    // ... one entry per reportable analyte
  }
}
```

COABuilder consumes `result_sources` as the source of truth — does not re-query SENAITE for these values. Mk1 has already resolved them and frozen them in the manifest. Builder's contribution is layout, formatting, and PDF generation; resolution is owned by Mk1.

### Migrations

All `IF NOT EXISTS`, all idempotent. Run via `database._run_migrations()`. No data-bearing migration on existing generations — they get the "manifest unavailable (pre-feature)" treatment in the UI.

### Activity log entries

```
event_type='coa_pin_changed'      payload={ analyte, old_pin, new_pin, reason }
event_type='coa_reportable_flipped' payload={ analysis_uid, reportable, reason }
event_type='coa_generated'        payload={ generation_number, source_changes: [...] }
event_type='coa_published'        unchanged from today
```

Existing `SampleActivityLog` query already filters by `parent_sample_id` so these surface on the parent's activity flyout without changes.

## Frontend

### New component: `<CoaSourcesCard>`

Lives in `src/components/senaite/coa/` (new directory). Used in `SampleDetails.tsx` for parent ARs. Reads `/api/coa/{parent}/sources` for the current draft; flips to `/api/coa/{parent}/sources/{generation_number}` when the user selects a historical generation in the existing COA history UI.

Skeleton:

```tsx
<Card>
  <SectionHeader icon={ScrollText} title="COA Sources">
    <CoaSourcesTable
      rows={data.rows}
      mode={isHistorical ? 'read-only' : 'editable'}
      onChangeSource={(analyte) => openPinDrawer(analyte)}
    />
    {hasUnresolved && (
      <Alert variant="warning">
        {unresolvedCount} analyte(s) need a source picked before this COA can generate.
      </Alert>
    )}
  </SectionHeader>
</Card>
```

### `<PinSourceDrawer>`

Reuses the existing `Dialog` + `DropdownMenu` primitives. Single form: candidate radio list + optional reason textarea + save/cancel. On save: PUT `/api/coa/.../pins/.../`; on success: invalidate `['coa-sources', parent]` query so the card refreshes; close drawer.

### Reportable toggle

A small "include in COA" checkbox/switch on the existing `AnalysisTable` rows on each sub-sample's detail page. When toggled off, the row gets a strikethrough + "excluded from COA" badge. PUT to `/api/analyses/.../reportable`.

The same toggle surfaces in the candidate list in the pin drawer (as a "this candidate is excluded" inline indicator + a "re-include" affordance for managers).

### History integration

When the existing COA-history dropdown (current behavior: lists prior generations) changes selection, the `<CoaSourcesCard>` re-fetches against the historical endpoint. URL state via the existing UI store key (`activeCoaGeneration` or whatever the current pattern is — confirm during impl).

## Edge cases

### Stale pin

A pin references `BW-0013-S03 / analysis_uid=abc`. The analysis is then withdrawn or its `reportable` flag flipped off. On next generate:

- Resolver finds no candidate matching the pin.
- Resolver does NOT fall back silently — it raises `stale_pin`.
- Panel surfaces "Identity-HPLC: pinned to BW-0013-S03 but that result is no longer available. Pick a new source." with the current candidate list.

This deliberately makes silence impossible.

### Pin to a not-yet-verified result

Permitted. Pin is saved. Generation blocks on `missing` (because the candidate is filtered out by the `state in (verified, published)` rule). Once the analysis is verified, generation auto-succeeds with the pin. This avoids forcing the manager to come back later.

### Sub-sample deleted after pin

`coa_result_pins.source_sample_id` is not FK'd to `lims_sub_samples.sample_id` — sub-sample deletion is rare but possible during intake mistakes. The pin becomes stale (see above). A nightly job (or a hook on sub-sample delete) can mark such pins with a `stale=true` flag if we want to flag them in the UI proactively; not in v1.

### Same analyte exists on two non-variance vials

E.g., HPLC ID run on both the parent AR and one HPLC sub. Both verified. Both reportable. No locked variance set. Resolver blocks on `needs_decision`. Manager pins one. No automatic "pick the parent" rule — the parent isn't inherently more authoritative; it's just another vial.

### Variance set summary as the source

When the parent has a locked variance set and the analyte is "variance-bearing" (HPLC purity, identity %, etc.), the resolver picks the **variance-set representative** as the COA source. Two choices for what "representative" means:

- **Mean across the locked set** — `result_value = mean(in-set candidates' values)`, frozen in the manifest. Manifest stores all contributing source_uids.
- **Manager-pinned representative** — manager pins one specific vial as the representative; resolver uses that vial's value.

v1 ships with the second option (manager picks the representative from within the locked set). The "compute mean" path is the open follow-on; see *Open questions §2*.

### Additional COAs (multi-customer COAs)

Existing `AdditionalCoaCard` infrastructure regenerates each additional COA independently. Each additional COA's manifest is its own `coa_generation_sources` row-set. Pins are shared across primary + additional COAs (one pin per parent × analyte); rationale: a manager who decides "BW-0013-S02 is the canonical Identity-HPLC source" means it for *all* customers' COAs.

If a future ask wants per-customer pin overrides, the pin table grows a `coa_config_id NULL` column (NULL = applies to primary + all additionals; non-NULL = specific additional). Not in v1.

## Phasing

**Phase 1 — Data model + resolver + auto path.**
- Three migrations.
- Resolver module.
- `/api/coa/{parent}/generate` calls the resolver; happy path (every analyte resolves to 1 candidate) generates as before but writes a `coa_generation_sources` manifest.
- No UI yet.
- Test: existing single-vial peptide order generates a COA with a manifest. Manifest rows match what was published.

**Phase 2 — COA Sources card (current-draft mode only).**
- `<CoaSourcesCard>` reads `/api/coa/.../sources`.
- Read-only view for now — no pin drawer.
- Shows the auto-resolved rows for draft generations.
- Useful even without pins because it answers "what's about to publish?" for the manager pre-generation.

**Phase 3 — Pin drawer + reportable toggle + blocking flow.**
- Pin upsert + delete endpoints.
- `<PinSourceDrawer>` UI.
- `analyses.reportable` migration + toggle in `AnalysisTable`.
- Resolver returns `blocking` decisions; generate returns 422; panel surfaces unresolved rows.
- End-to-end test: variance HPLC scenario — manager pins, regen succeeds.

**Phase 4 — Historical mode + activity log + manifest snapshots.**
- Activity log entries fire on pin changes / reportable flips / generate / publish.
- Historical mode of the panel reads from `coa_generation_sources` + the snapshot column.
- COA history dropdown wired to switch the panel's data source.

**Phase 5 — Variance-set integration.**
- Locked variance set → `mode='variance_set'` auto-resolution.
- Manager picks the variance-set representative inline in the panel.
- Variance Addon spec (separate) consumes the same manifest rows for its inserts on the PDF.

Phases 1–3 are the MVP — that's where the user-visible behavior change happens. Phases 4–5 are polish / contract for downstream work.

## Open questions

### 1. Default-pin policy when a sub result is verified

When a tech verifies the first result for an analyte on a sub-sample (and zero other candidates exist), nothing's ambiguous and we auto-resolve. **Question:** when a *second* candidate later joins (e.g. a variance run on another vial gets verified), do we:

- **(a)** Auto-promote the first-verified result to a pin so the COA generation still doesn't prompt the manager. The manager only sees the override panel if they choose to.
- **(b)** Leave the row unpinned. As soon as the second candidate verifies, the next generation attempt blocks until the manager picks.

I lean **(a)** — fewer interruptions, audit trail intact via "auto-pinned on first verify" activity entry. (b) is more conservative but feels like nagging the manager about decisions they've already implicitly made by setting up the variance run.

### 2. Aggregation mode (`variance_set` with computed value)

v1 ships variance-set integration as "manager picks the representative vial inside the locked set." The natural follow-on is "compute the mean (or median, or geometric mean) across the locked set and report THAT value." That requires the variance addon to also write the computed value back somewhere COABuilder can read it.

Two paths:
- Pin row of `mode='variance_set'` carries an `aggregation_fn` column (`mean | median | none`). Resolver computes at generation time. Manifest freezes the computed value.
- Variance set itself stores the "reportable value" (computed on lock or recomputed on result-edit-while-unlocked). Pin just references "use the variance set."

The second is cleaner because the value lives with the set it derives from. Worth nailing down before Phase 5.

### 3. Where the `reportable` flag lives

The cleanest spot is a new column on `analyses`. If that's a SENAITE-mirrored cache we'd be adding a column to a table that gets refreshed from SENAITE — risk of being clobbered. Two options:

- Add column, exclude from the SENAITE refresh mapping (most refreshes are column-scoped already, so likely safe).
- New Mk1-local sidecar table `analysis_reportable (analysis_uid PK, reportable bool, ...)` joined at read time.

Need to confirm the actual schema lifecycle of `analyses` before committing. (Backend reviewer / Forrest knows this offhand.)

### 4. Pin scope: per-customer (additional-COA) or shared?

Currently spec'd as shared. Most multi-customer COA setups want the same source result going out under different branding — shared is the right default. A future per-customer override is a column add on the pin table, not a redesign.

### 5. How much of the existing COA history UI changes

The existing flow: dropdown of "v1 / v2 / v3" generations on `PublishedCOACard`. Plan is to have selecting a historical generation flip the COA Sources card into historical mode. Two ways to wire this:

- COA Sources card subscribes to the same `activeCoaGeneration` UI-store key that the COA history dropdown writes.
- Drilling: parent passes `selectedGenerationNumber` down to both components.

Either works. Flag during impl.

## Verification

### Phase 1 acceptance

- New single-vial peptide check-in → generate COA → `coa_generation_sources` has one row per reported analyte, all `resolution_mode='auto'`, `candidates_count=1`.
- Regen the same COA → new generation rows, prior generation's manifest unchanged.
- Existing peptide-request E2E test passes (no behavior change for single-vial).

### Phase 3 acceptance

- Variance HPLC: parent + 2 HPLC subs, all 3 verified, no variance lock. Attempt generate → 422 with `unresolved=['IDENTITY_HPLC', 'PURITY_HPLC', ...]`. UI shows blocked rows.
- Manager pins each analyte → generate succeeds → manifest rows have `resolution_mode='pin'`, `candidates_count=3`.
- Regen-and-republish flow: change a pin, regen → new manifest, prior frozen.

### Phase 4 acceptance

- Open historical generation in the COA Sources card → shows frozen rows even after the source analysis's value is later edited in SENAITE.
- Activity log on the parent shows "Forrest pinned Identity-HPLC to BW-0013-S02 — reason: 'S02 was the clean run'" with timestamp + user.

### Phase 5 acceptance

- Locked variance set with all members verified → COA generates without prompting; manifest rows for variance-bearing analytes have `resolution_mode='variance_set'`.
- Unlocking the variance set after publish does NOT alter the prior manifest (immutability check).
